1. 무료 모델만 사용해야 한다면 품질을 어떻게 유지할 것인가?
1.1 문제 상황
무료 모델만 사용해야 하는 경우, 유료 고성능 모델에 비해 다음과 같은 한계가 발생할 수 있다.
긴 논문 초록이나 본문을 한 번에 처리할 때 문맥을 놓칠 수 있다.
복잡한 지시를 동시에 주면 출력 형식이 흔들릴 수 있다.
원문 사실과 연구 활용을 위한 추론이 섞일 수 있다.
논문에 없는 표본 수, 분석기법, 통계모형, 구체적 수치를 임의로 생성할 위험이 있다.
따라서 무료 모델을 사용할 때는 한 번에 모든 작업을 맡기기보다, 작업을 단계별로 분리하고 각 단계의 출력 기준을 좁게 설정하는 방식이 필요하다.
---
1.2 대응 전략: 단계 분리형 프로세스
본 프로젝트의 과업은 기술경영/CFM 논문 요약 및 연구 활용성 분석이다.  
무료 모델을 사용할 경우 다음과 같이 작업을 4단계로 나누어 수행한다.
단계	목적	출력 내용
1단계	원문 사실 추출	연구 질문, 연구 설계, 측정 연도, 성과 변수, 원문에 명시된 수치
2단계	누락 정보 확인	표본 수, 산업, 국가, 분석기법, 통계모형의 명시 여부
3단계	연구 활용성 분석	CFM 연구와의 관련성, 측정 방식 차별화, 후속 연구 가능성
4단계	최종 요약 통합	5개 출력 항목으로 정리하고 원문 사실과 추론 분리
이 방식은 무료 모델이 긴 문맥을 한 번에 처리하지 않아도 되도록 하며, 각 단계에서 확인해야 할 내용을 명확히 제한한다.
---
1.3 무료 모델용 프롬프트 예시
1단계: 원문 사실만 추출
```text
아래 논문 초록에서 원문에 명시된 사실만 추출해줘.

출력 항목:
1. 연구 질문
2. 연구 설계
3. 측정 연도
4. 성과 변수
5. 원문에 명시된 수치

주의:
- 원문에 없는 정보는 추측하지 말 것.
- 원문에 없으면 "원문에 명시 없음"이라고 표시할 것.
```
2단계: 누락 정보 확인
```text
아래 항목이 원문 초록에 명시되어 있는지 확인해줘.

확인 항목:
1. 표본 수
2. 산업
3. 국가
4. 분석기법
5. 통계모형

출력 규칙:
- 명시되어 있으면 원문 근거와 함께 작성
- 명시되어 있지 않으면 "원문에 명시 없음"으로 표시
```
3단계: 연구 활용성 분석
```text
앞서 추출한 원문 사실을 바탕으로 CFM 연구와의 관련성을 제안해줘.

주의:
- 원문에 직접 있는 내용은 "원문 사실"로 표시할 것.
- 연구 활용을 위한 해석은 반드시 [추론]으로 표시할 것.
- 원문에 없는 표본 수, 분석기법, 수치는 생성하지 말 것.
```

1.4 기대 효과
무료 모델 사용 시 위험	대응 방식
긴 문맥 처리 실패	작업을 4단계로 분리
출력 형식 불안정	각 단계의 출력 항목을 제한
원문 사실과 추론 혼합	“원문 사실”과 “[추론]”을 표로 분리
환각 발생	원문에 없는 정보는 “원문에 명시 없음”으로 처리
따라서 무료 모델만 사용할 경우에도 사실 추출 → 누락 정보 확인 → 연구 활용성 분석 → 최종 요약 순서를 유지하면 품질을 일정 수준 이상으로 관리할 수 있다.

2. 업무 정책, 규정, 수치가 자주 바뀐다면 환각을 어떻게 줄일 것인가?
2.1 문제 상황
업무 정책, 규정, 수치가 자주 바뀌는 환경에서는 모델의 일반지식이나 기억에 의존하면 환각 위험이 커진다.
예를 들어 다음과 같은 정보는 시간이 지나면 바뀔 수 있다.
내부 정산 규정
지원금 지급 기준
신청 마감일
법률 조항
회계 기준
비용 한도
평가 기준
조직 내부 정책
따라서 이런 업무에서는 모델에게 “알고 있는 대로 답하라”고 지시하면 안 된다.  
반드시 사용자가 제공한 최신 기준 문서 안에서만 답하도록 제한해야 한다.
---
2.2 대응 전략: 기준 문서 기반 답변
정책, 규정, 수치가 자주 바뀌는 업무에서는 다음 원칙을 적용한다.
운영 원칙	설명
최신 문서 입력	공식 규정, 내부 정책 문서, 최신 공지, 기준일이 있는 자료를 입력으로 제공
기준일 표시	답변에 문서명과 기준일을 함께 표시
근거 위치 표시	가능하면 조항, 문단, 표 번호를 함께 표시
문서 외 정보 생성 금지	문서에 없는 예외 조건, 금액, 날짜, 담당자는 생성하지 않음
사람의 최종 검토	법률, 인사, 회계, 비용 업무는 담당자가 최종 확인
---
2.3 정책/규정 업무용 프롬프트 예시
```text
아래 정책 문서만 근거로 답변해줘.

규칙:
1. 문서에 있는 내용만 사실로 말할 것.
2. 문서에 없는 내용은 "문서에 명시 없음"이라고 표시할 것.
3. 금액, 날짜, 조항, 예외 조건은 절대 추측하지 말 것.
4. 답변에는 기준 문서명과 기준일을 함께 표시할 것.
5. 최종 판단이 필요한 내용은 "담당자 확인 필요"로 표시할 것.

[기준 문서명]
[기준일]
[문서 내용]
```

즉, 정책이나 수치가 자주 바뀌는 업무에서는 모델을 최종 판단자가 아니라 문서 기반 요약 및 검증 보조자로 운영해야 한다.

3. 10턴 이상 대화에서 문맥이 깨지면 어떻게 개선할 것인가?
3.1 문제 상황
10턴 이상 대화에서는 초기에 설정한 조건이 뒤쪽 대화에서 약해질 수 있다.
본 프로젝트에서 유지되어야 하는 핵심 조건은 다음과 같다.
5개 출력 항목 유지
원문에 없는 정보 생성 금지
`[추론]` 태그 유지
표본 수, 분석기법, 통계모형은 원문에 없으면 “원문에 명시 없음”으로 처리
CFM 연구와의 관련성은 원문 사실과 추론을 분리
대화가 길어지면 모델이 일부 조건을 잊거나, 사용자의 최신 요청만 반영하다가 기존 안전장치를 놓칠 수 있다.
---
3.2 대응 전략 1: 고정 규칙과 변경 요청 분리
문맥 단절을 줄이기 위해 프롬프트를 고정 규칙과 변경 가능한 요청으로 분리한다.
고정 규칙
```text
[고정 규칙]
1. 출력은 항상 5개 항목을 유지한다.
2. 원문에 없는 정보는 생성하지 않는다.
3. 추론은 반드시 [추론]으로 표시한다.
4. 표본 수, 산업, 분석기법, 통계모형은 원문에 없으면 "원문에 명시 없음"으로 처리한다.
5. 원문 사실과 연구 활용 제안은 분리한다.
```
변경 가능한 요청
```text
[변경 가능한 요청]
- 더 짧게 작성
- 표로 변환
- CFM 관련성 강화
- 연구계획서용 문장으로 변경
- 원문 사실과 추론 분리
```
이렇게 분리하면 사용자가 중간에 출력 길이나 형식을 바꾸더라도 핵심 안전장치는 유지된다.
---
3.3 대응 전략 2: 중간 상태 요약
긴 대화에서는 3~4턴마다 현재 조건을 요약하게 한다.
```text
현재까지 유지해야 할 조건을 요약해줘.

1. 5개 출력 항목 유지
2. 원문에 없는 정보 생성 금지
3. 추론은 [추론] 표시
4. 표본 수와 분석기법은 원문에 없으면 명시 없음 처리
5. 원문 사실과 연구 활용 제안 분리
```
이 방식은 모델이 앞선 조건을 다시 인식하게 하여 문맥 이탈 가능성을 줄인다.
---
3.4 대응 전략 3: 최종 검증 체크리스트
최종 답변을 내기 전에는 아래 체크리스트를 사용한다.
검증 항목	확인 기준
출력 형식	5개 항목이 유지되었는가
사실성	원문에 없는 정보가 생성되지 않았는가
추론 분리	`[추론]` 태그가 유지되었는가
누락 정보 처리	표본 수, 분석기법 등이 “원문에 명시 없음”으로 처리되었는가
조건 변경 반영	최신 요청은 반영하되 기존 안전장치는 유지했는가

```text
[현재 사용자 요청]

- [추론] 태그 유지 여부
- 누락 정보 처리 여부
```

4. 심층 인터뷰 답변 요약
Q1. 무료 모델만 써야 한다면 어떻게 할 것인가?
무료 모델은 긴 문맥과 복잡한 지시를 한 번에 처리하는 능력이 약할 수 있으므로 작업을 사실 추출 → 누락 정보 확인 → 연구 활용성 분석 → 최종 요약으로 나누겠습니다. 또한 원문 사실과 `[추론]`을 표로 분리하게 하여 환각을 줄이겠습니다.
Q2. 정책이나 수치가 자주 바뀐다면 어떻게 할 것인가?
모델의 기억에 의존하지 않고 최신 기준 문서를 입력으로 제공하겠습니다. 답변에는 문서명, 기준일, 조항 또는 문단을 함께 표시하게 하고, 문서에 없는 내용은 “확인 필요”로 처리하겠습니다. 모델은 최종 판단자가 아니라 검증 보조자로 사용하겠습니다.
Q3. 10턴 대화에서 문맥이 깨진다면 어떻게 할 것인가?
프롬프트를 고정 규칙과 변경 가능한 요청으로 분리하겠습니다. 5개 항목 유지, 원문에 없는 정보 생성 금지, `[추론]` 태그 사용 같은 규칙은 고정하고, 길이 축소나 표 변환 같은 요청만 변경 가능하게 하겠습니다. 또한 중간 상태 요약과 최종 검증 체크리스트를 넣어 문맥 이탈을 줄이겠습니다.
