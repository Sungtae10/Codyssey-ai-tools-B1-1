1. 무료 모델만 사용해야 한다면 품질을 어떻게 유지할 것인가?

무료 모델만 사용해야 한다면, 고성능 유료 모델에 한 번에 긴 작업을 맡기는 방식 대신 작업을 단계별로 분리하는 프로세스를 사용한다.

현재 과업은 “기술경영/CFM 논문 요약 및 연구 활용성 분석”이다. 이 과업에서 무료 모델을 사용할 경우, 긴 논문 텍스트를 한 번에 입력하면 문맥 누락, 출력 형식 불안정, 추론과 사실의 혼합이 발생할 수 있다. 따라서 다음과 같이 작업을 나눈다.

무료 모델을 사용할수록 프롬프트는 더 짧고 명확해야 하므로, 입력 템플릿을 엄격하게 유지한다. 특히 “원문 사실”, “원문에 명시 없음”, “[추론]”을 표로 분리하게 하면 무료 모델에서도 결과 품질을 일정하게 유지할 수 있다.

2. 업무 정책, 규정, 수치가 자주 바뀐다면 환각을 어떻게 줄일 것인가?

업무 정책, 규정, 수치가 자주 바뀌는 환경에서는 모델의 일반지식이나 기억에 의존하면 환각 위험이 커진다. 따라서 모델이 자체 지식으로 답하지 못하게 하고, 반드시 최신 기준 문서 안에서만 답하도록 운영해야 한다.

본 프로젝트의 논문 요약 과업에서는 원문 초록을 기준 문서로 사용했다. 이와 같은 원칙을 정책/규정/수치 업무에 적용하면 다음과 같다.

최신 기준 문서를 입력으로 제공
공식 규정집
내부 정책 문서
최신 공지
날짜가 명시된 자료
숫자 또는 기준 변경 이력이 있는 문서
출처와 기준일 표시
답변에 기준 문서명, 날짜, 조항 또는 문단을 표시하게 한다.
예: “2026년 6월 기준 내부 규정 문서 3조에 따르면…”
문서에 없는 내용은 생성 금지
정책 문서에 없는 예외 조건, 금액, 날짜, 담당 부서는 만들지 않는다.
없으면 “문서에 명시 없음” 또는 “확인 필요”라고 표시한다.
정책 변경 가능성 표시
규정이나 수치가 자주 바뀌는 항목은 “최신 문서 확인 필요”를 함께 표시한다.
사람의 최종 검토 단계 유지
모델은 최종 판단자가 아니라 초안 작성자와 검증 보조자로 사용한다.
특히 법률, 인사, 회계, 비용 정산처럼 오류 비용이 큰 업무는 담당자의 최종 확인을 거친다.

따라서 정책이나 수치가 자주 바뀌는 업무에서는 “모델이 알고 있는 답”을 묻는 것이 아니라, “사용자가 제공한 최신 문서 안에서만 답하게 하는 방식”으로 환각을 줄일 수 있다.

3. 10턴 이상 대화에서 문맥이 깨지는 문제가 생기면 어떻게 개선할 것인가?

10턴 이상 대화에서는 초기에 설정한 조건이 뒤쪽 대화에서 약해질 수 있다. 예를 들어 본 프로젝트에서는 다음 조건들이 유지되어야 한다.

5개 출력 항목 유지
원문에 없는 정보 생성 금지
[추론] 태그 유지
표본 수, 분석기법, 통계모형은 원문에 없으면 “원문에 명시 없음” 처리
기술경영/CFM 연구와의 관련성은 원문 사실과 추론을 분리

문맥이 깨지는 문제가 발생하면 프롬프트 구조를 다음과 같이 개선한다.

1) 고정 규칙과 변경 요청을 분리한다

프롬프트를 다음 두 영역으로 나눈다.

고정 규칙

출력은 5개 항목으로 유지
원문에 없는 정보는 생성 금지
추론은 반드시 [추론]으로 표시
표본 수, 산업, 분석기법, 통계모형은 원문에 없으면 “원문에 명시 없음”으로 처리

변경 가능한 요청

더 짧게 작성
표로 변환
CFM 관련성 강화
연구계획서용 문장으로 바꾸기
원문 사실과 추론 분리

이렇게 분리하면 사용자가 중간에 톤이나 형식을 바꾸더라도 핵심 안전장치는 유지된다.

2) 중간 상태 요약을 사용한다

긴 대화에서는 3~4턴마다 AI에게 현재 조건을 요약하게 한다.

예시:

현재까지 유지해야 할 조건을 요약해줘.
1. 5개 출력 항목 유지
2. 원문에 없는 정보는 생성 금지
3. 추론은 [추론] 표시
4. 표본 수와 분석기법은 원문에 없으면 명시 없음 처리

이렇게 하면 모델이 앞선 조건을 다시 인식하여 문맥 단절 가능성을 줄일 수 있다.

3) 최종 출력 전 검증 체크리스트를 추가한다

최종 답변을 내기 전에 다음 기준을 확인하게 한다.

검증 항목	확인 기준
출력 형식	5개 항목이 유지되었는가
사실성	원문에 없는 정보가 생성되지 않았는가
추론 분리	[추론] 태그가 유지되었는가
누락 정보 처리	표본 수, 분석기법 등 없는 정보가 “원문에 명시 없음”으로 처리되었는가
조건 변경 반영	사용자의 최신 요청만 반영하고 기존 안전장치는 유지했는가

이와 같이 프롬프트를 “고정 규칙 + 변경 요청 + 상태 요약 + 최종 검증” 구조로 개선하면 10턴 이상 대화에서도 문맥 유지 성능을 높일 수 있다.

4. 심층 인터뷰 답변 요약

대면 평가에서 짧게 답해야 할 경우 다음과 같이 설명할 수 있다.

Q1. 무료 모델만 써야 한다면?

무료 모델은 긴 문맥과 복잡한 지시를 한 번에 처리하는 능력이 약할 수 있으므로, 작업을 “사실 추출 → 누락 정보 확인 → 연구 활용성 분석 → 최종 요약”으로 나누겠습니다. 또한 원문 사실과 [추론]을 표로 분리하게 하여 환각을 줄이겠습니다.

Q2. 정책이나 수치가 자주 바뀐다면?

모델의 기억에 의존하지 않고, 최신 기준 문서를 입력으로 제공하겠습니다. 답변에는 문서명, 기준일, 조항 또는 문단을 함께 표시하게 하고, 문서에 없는 내용은 “확인 필요”로 처리하겠습니다. 모델은 최종 판단자가 아니라 검증 보조자로 사용하겠습니다.

Q3. 10턴 대화에서 문맥이 깨진다면?

프롬프트를 고정 규칙과 변경 가능한 요청으로 분리하겠습니다. 5개 항목 유지, 원문에 없는 정보 생성 금지, [추론] 태그 사용 같은 규칙은 고정하고, 길이 축소나 표 변환 같은 요청만 변경 가능하게 하겠습니다. 또한 중간 상태 요약과 최종 검증 체크리스트를 넣어 문맥 이탈을 줄이겠습니다.
